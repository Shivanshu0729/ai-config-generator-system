The compiler pipeline

A single prompt is rejected. Every input passes through four isolated, validated stages — like a real compiler.

01

Intent extraction

Parses your description into a typed intermediate form — app type, entities, roles, features, and ambiguities. Vague inputs trigger clarification requests.

→ intent.json
02

System design

Converts intent into architecture — pages with routes, backend services, data flows, and third-party integrations. Defines the full system shape.

→ architecture.json
03

Schema generation

Four parallel sub-compilers generate UI schema, REST API schema, database schema, and auth/roles config simultaneously against strict contracts.

→ ui · api · db · auth
04

Validation + repair

Cross-layer consistency checks: API fields match DB schema, UI routes match auth config, roles match intent. Mismatches are auto-repaired, not silently dropped.

→ validated_config.json

What makes this a compiler, not a script

        
Each requirement from the task specification is implemented as a distinct, testable system component.

      

      

        

          

            
          

          
Multi-stage pipeline

          
Four isolated stages each with their own prompt contract, output schema, and retry budget. No stage shares context with another until explicitly passed through.

        

        

          

            
          

          
Intelligent repair

          
Mismatches between layers are detected and surgically fixed — missing roles injected, routes synced, premium flags corrected. No blind full-retry.

        

        

          

            
          

          
Strict schema contracts

          
Every stage has a defined JSON output contract. Invalid JSON, missing fields, and type errors are caught — the stage retries rather than passing bad data forward.

        

        

          

            
          

          
Failure handling

          
Vague or conflicting inputs are caught at Stage 1 before wasting API calls. The system asks a targeted clarification question instead of producing garbage output.

        

        

          

            
          

          
Parallel sub-compilers

          
Stage 3 fires four independent API calls simultaneously — UI, API, DB, and Auth schemas are generated in parallel, then reconciled in Stage 4.

        

        

          

            
          

          
Live metrics

          
Every run tracks latency, API retry count, and quality score (ratio of passing validation checks). Execution-level observability, not just output.
